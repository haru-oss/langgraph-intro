以下の設計原則とルールに従い、指定された言語・ライブラリの学習教材を作成してください。

# 教材作成の設計原則

## 1. 全体構成: 3層構造

教材は必ず以下の3層で構成する。

### 層1: 基礎編（training）
- 対象ライブラリの**本番で必要な基本機能**を1トピック1ファイルで学ぶ
- 番号付きフォルダ（`00_`, `01_`, ...）で学習順序を明示
- 各トピックは独立して動作する最小限のコード（目安: 30-80行）

### 層2: 応用編（advance）
- 基礎を前提とした高度な機能やパターンを学ぶ
- 番号は基礎編から通し番号で継続する（基礎が00-05なら応用は06-）
- 実務での応用例を必ず含める

### 層3: 実践編（practice）
- 基礎編で学んだ機能を**複数組み合わせた**課題を提示する
- 実装コードは提供しない（学習者が自分で実装する演習）
- README に設計書（ファイル構成、処理フロー、State/型設計）を詳細に記載し、学習者の設計指針とする
- 「これは演習課題です」と明記する

---

## 2. トピック選定の基準

「面白い機能」ではなく「本番運用で避けて通れない機能」を優先する。
選定時に以下の問いを立てる:

1. このライブラリを使ってWebアプリ/サービスを作るとき、この機能なしで済むか？
2. 実務で「これ知らなくてハマった」となりやすい機能か？
3. 公式ドキュメントだけでは理解しづらい概念か？

---

## 3. 各トピックフォルダの構成

```
XX_TopicName/
  README.md       # 解説ドキュメント（後述のフォーマットに従う）
  topic_bot.py    # 最小限の動作するサンプルコード
```

---

## 4. README.md の書き方ルール

### 4.1 冒頭: 一文要約
最初の1文で「このコードは何を実現するものか」を明確にする。

良い例: 「このコードは、**『AIが会話の内容を記憶し、ユーザーごとに区別して会話する』**という機能（永続化）を実現するための最小限のサンプルです。」

### 4.2 「なぜ必要か」を先に書く
コードの解説に入る前に、**実務でなぜこの機能が必要なのか**を具体的な業務シーン付きで説明する。

```
学ぶべきこと: （技術的なキーワード）
なぜ必要か: （実務での具体的な課題・シーン）
```

### 4.3 比喩・たとえを使う
技術概念を日常の体験に例える。1トピックにつき最低1つは比喩を入れる。

良い例:
- 「thread_id は会員番号のようなもの」
- 「通常通信 = ラーメン屋（全部完成してから出てくる）、ストリーミング = わんこそば（できた分から出てくる）」

### 4.4 行番号付き解説
コードの解説は「## N. セクション名 (XX-YY行目)」の形式で、対応するコードの行範囲を明記する。

```markdown
## 1. 準備フェーズ (1-16行目)
## 2. グラフの組み立て (21-25行目)
## 3. 記憶の仕組みを導入 (27-32行目) ★ここが一番重要
```

最も重要な箇所には「★ここが一番重要」を付けて、学習者の注意を引く。

### 4.5 コード解説にはインラインコードを含める
解説文中に実際のコード片を引用し、その直下で「何をしているか」「なぜそう書くか」を説明する。

### 4.6 まとめセクション
各READMEの末尾に「まとめ」を置き、学んだことの本質を1-2文で要約する。
実務での応用アイデアや拡張アイデアも含める。

### 4.7 実務応用例
応用編では、学んだパターンを使った具体的なシステム例を3つ以上示す。
ASCII図で構成を示すと効果的。

```
supervisor
  |-> faq_agent
  |-> technical_support_agent
  |-> billing_agent
```

## 5. サンプルコードの書き方ルール

### 5.1 最小限の原則
- 1ファイルで完結する（import以外の外部依存なし）
- そのトピックに関係ないコードは書かない
- 目安: 30-80行（100行を超えたら分割を検討）

### 5.2 コメントは構造を示す程度に
コード内のコメントは「# N. セクション名」の形式で構造を示す程度にとどめる。
詳細な説明はREADMEに書く。コード側は読みやすさを優先。

```python
# 1. Stateの定義
class State(TypedDict):
    messages: Annotated[list, add_messages]

# 2. ノードの定義
def chatbot(state: State):
    return {"messages": [llm.invoke(state["messages"])]}

# 3. グラフの構築
builder = StateGraph(State)
```

### 5.3 実行確認できるmainブロックを含める
ファイル末尾に `if __name__ == "__main__":` ブロックを置き、
`python topic_bot.py` で即座に動作確認できるようにする。

### 5.4 出力にはプレフィックスを付ける
実行時の出力には `[NodeName]` のようなプレフィックスを付け、
どの処理がどの出力を生んだか追跡できるようにする。

```python
print("[Generator] 初稿を作成中...")
print("[Reflector] 批評中...")
```

## 6. ルートREADME

プロジェクトルートのREADMEには以下を含める:

1. **プロジェクト構成**: ディレクトリツリー（tree形式）
2. **学習の流れ**: ステップ1（基礎）→ ステップ2（応用）→ ステップ3（実践）の順序と各トピック一覧
3. **セットアップ手順**: clone、仮想環境、pip install、環境変数の設定を具体的なコマンド付きで
4. **環境変数テンプレート**: `.env.example`の内容を記載（APIキーは`your_xxx_here`）
5. **必要な環境**: 言語バージョン、必要なAPIキー等
6. **参考リンク**: 公式ドキュメント、関連記事（最大5つ程度）
7. **学習のコツ**: 5つ程度のアドバイス

---

## 7. ライブラリ解説

ルートREADMEまたは専用ファイル**1箇所のみ**に、requirements.txt（または同等のファイル）の全ライブラリの役割一覧を記載する。
各ライブラリについて以下を書く:

- 名前とバージョン
- 一言の役割説明
- なぜこのプロジェクトで必要か

重複して複数箇所に書かない。他のREADMEからはリンクで参照する。

---

## 8. ファイル命名と構造の統一

- フォルダ名: `XX_TopicName`（XX=通し番号2桁、TopicName=PascalCase）
- ソースファイル名: `topic_name_bot.py`（snake_case + `_bot`サフィックス）（言語に適した命名に変更可）
- 仮想環境: プロジェクトルートに1つの`.venv`にまとめる（基礎/応用で分けない）
- 環境変数: `.env.example`をコミットし、`.env`は`.gitignore`に含める

---

## 9. 進行中・未実装の表示

未実装のトピックやフォルダがある場合:
- READMEに「（計画中）」「（未実装 - 演習課題）」と明記する
- 実装済みと未実装を区別できるチェックボックスリストを使う

```markdown
- [x] 00_Introduction
- [x] 01_Persistence
- [ ] 07_Parallel（計画中）
```

---

## 10. 避けるべきこと

- READMEの内容を複数ファイルにコピペしない（1箇所に書いてリンクで参照）
- 1ファイルに複数トピックを詰め込まない
- 抽象的な説明だけで終わらせない（必ず具体例を付ける）
- テスト環境なしで「動作確認しましょう」と書かない（期待出力例を示す）
- コードにREADMEと同じ長さのコメントを書かない（説明はREADMEに任せる）

---

# 使い方

教材を作成する際は、以下の情報を指定してください:

1. **対象の言語/ライブラリ**: 何を教える教材か
2. **想定する学習者**: 前提知識のレベル
3. **トピック一覧**（あれば）: 基礎編・応用編で扱いたいトピック
4. **実践課題**（あれば）: practice で作らせたいもの

指定がない項目については、上記の設計原則に基づいて提案します。
